# Maturity

This document states which parts of Sweetspot Modular Architecture work today and which still need field proof. Adopters, maintainers, and public communicators need it before relying on or describing the framework. Read it when assessing release risk, planning validation, or writing public claims. Remember to match every maturity claim to current evidence rather than the intended roadmap.

Current status: alpha working framework.

This means:

- the philosophy is battle-tested in AI-heavy work
- the schema and tools are active
- the registry format can change
- canonical status should be conservative
- public claims should be humble and evidence-based

## What Is Solid

- brick manifest concept
- registry and wiki flow
- model/human/tool provenance
- canonical blocker rules
- minimum responsible code rule
- validation and security gate scripts
- agent skill packaging

## What Still Needs Real-World Proof

- clone success metrics across many projects
- duplicate detection quality
- CI adoption in production repos
- security gate false-positive tuning
- model provenance capture from real agent runners
- canonical promotion history

## Public Claim Level

Safe claim:

> SMA is a working framework for reliability-oriented, AI-assisted module reuse.

Unsafe claim:

> SMA is the architecture everyone should use.
